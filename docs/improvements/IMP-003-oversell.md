# IMP-003 · 재고 초과판매(oversell) — DB 조건부 UPDATE로 원자화

- 슬라이스: `S04`
- 날짜: 2026-07-06
- 유형: 정량(동시성 테스트로 측정) — 동시성·정합성
- 관련 커밋: `6b74a7e` (feat: 재고 코어)
- 벤치 파일: [`benchmarks/seat-oversell-before.json`](../../benchmarks/seat-oversell-before.json)
  → [`benchmarks/seat-oversell-after.json`](../../benchmarks/seat-oversell-after.json)

> 흐름: 문제 정의 → 증상(수치) → 가설 → 도구로 검증 → 해결 → 재측정 → 한계.

## 1. 상황 (Context)
선착순 예매의 핵심은 "재고 1석에 2명이 동시에 결제 진입"할 때 **한 명만** 잡는 것.
좌석 상태(AVAILABLE/HELD/SOLD)는 우리 DB(seats)에 있고, 선점은 그 상태를 바꾸는 일이다.

> **IMP-004(대기열)/IMP-007(발급)과 대비**: 이건 재고를 다투는 **좌석 선점**으로, 선착순의
> 최종 정확성이 걸린 지점이다. (대기열이 앞단에서 트래픽을 정원 단위로 줄여주지만, 재고 자체의
> 원자성이 없으면 결국 초과판매가 난다.)

## 2. 문제 정의 + 분류
선점을 "상태 조회 → AVAILABLE이면 → HELD로 변경"으로 나누면(check-then-act) 동시 실행 시
같은 좌석을 여럿이 잡는 **초과판매(oversell)** 가 난다.

- 계층 분류: **동시성**(check-then-act 레이스). 상태가 DB에 있으므로 원자성도 DB에서.
- 왜: 두 트랜잭션이 거의 동시에 같은 좌석을 AVAILABLE로 읽고, 각자 HELD로 UPDATE.

## 3. 증상 (측정된 증거)
- 측정: `SeatInventoryIntegrationTest` — 1석에 20스레드가 동시에 선점(레이스 창 20ms, JdbcTemplate).
- before(비원자):
  | 지표 | 값 |
  |------|----|
  | 좌석 | 1 |
  | 동시 시도 | 20 |
  | 선점 성공 | **20** |
  | **초과판매(oversell)** | **19** |

## 4. 가설 (검증 전)
- 여러 스레드가 같은 좌석을 AVAILABLE로 읽은 뒤 각자 UPDATE → 1석이 여러 번 팔린다.

## 5. 검증 (도구로 확인 → 확정 원인)
- 도구: 동시성 통합 테스트(Testcontainers Postgres). 비원자 `읽고→무조건 UPDATE`를 20스레드로 재현.
- 결과: 1석에 20건 선점 성공 → check-then-act 레이스로 초과판매 확정.

## 6. 조치 (해결)
선점을 **DB 조건부 UPDATE 단일 문장**으로 원자화:
```sql
UPDATE seats SET status='HELD', updated_at=now()
WHERE id IN (:ids) AND status='AVAILABLE'
```
→ **영향 행 수 == 요청 좌석 수**를 검증. 다르면 롤백 + `SOLD_OUT`. DB 행 락이 동시 트랜잭션
중 하나만 통과시키므로 초과판매가 원천 불가. (JPA `@Modifying`으로 구현, ADR-003.)

## 7. 결과 (재측정 — 동일 조건)
| 지표 | before | after | 변화 |
|------|--------|-------|------|
| 선점 성공 | 20 | **1** | 정확 |
| **초과판매(oversell)** | 19 | **0** | **-100%** |

→ 1석 20동시 시도에도 정확히 1명만 성공. `조건부UPDATE_동시선점은_초과판매되지_않는다`로 보장.

## 8. 트레이드오프 / 한계 / 다음 개선
- **왜 락/Redis가 아닌 조건부 UPDATE인가**: 비관락은 락 경합·데드락, 낙관락(@Version)은 재시도
  폭증, Redis 카운터는 좌석 식별을 이중 관리. 조건부 UPDATE는 단일 문장이 곧 원자 연산 + 좌석
  식별 그대로 + 재시도 불필요 → 단순·정확(ADR-003).
- **부분 선점**은 전부-or-전무(요청 3석 중 1석 매진 시 전체 롤백 + SOLD_OUT).
- 데모 규모(등급별 수십 석)에 최적. 특정 공연이 초고 TPS면 그 이벤트만 Redis 재고 카운터로
  핫패스 분리 검토(그때 대용량 시드 후 p95 비교를 별도 IMP로).

## 9. 배운 점 (면접 답변용)
> 선착순의 핵심은 재고 정확성인데, "상태 확인 → 변경"으로 나누면 동시 선점에서 1석이 여러 번
> 팔리는 초과판매가 납니다. 1석에 20스레드를 던져 20건이 팔리는 걸 재현한 뒤, 상태가 DB에 있으니
> 원자성도 DB에서 강제하는 게 맞다고 보고 `WHERE status='AVAILABLE'` 조건부 UPDATE로 바꿔
> 초과판매를 0으로 만들었습니다. 애플리케이션 if문이 아니라 DB 원자 연산으로 정원을 지킨 사례이며,
> 대기열(Redis Lua)·발급(SET NX)과 같은 "상태 있는 계층에서 원자화" 원칙을 재고에도 일관 적용했습니다.
