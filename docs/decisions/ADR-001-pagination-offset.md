# ADR-001 · 목록 페이지네이션은 offset을 기본으로 한다

- 상태: Accepted
- 날짜: 2026-07-01
- 슬라이스: S02(공연 목록/검색). 이후 목록 화면 전반에 적용.
- 관련: [[event]], `/search` 페이지네이션

> ADR은 IMP(문제→개선→측정)와 다르다. "무엇을, 왜 그렇게 결정했고, 언제 뒤집는가"를 남긴다.

## 맥락 (Context)
공연 목록·검색 결과를 페이지 단위로 조회해야 한다. 방식은 크게 두 가지:
- **offset**: `LIMIT size OFFSET page*size` + `count(*)` — 페이지 번호로 임의 이동, 전체 페이지 수 표시.
- **cursor(keyset)**: `WHERE (정렬키) > 마지막본값 LIMIT size` — 마지막 위치부터 이어 읽기.

## 결정 (Decision)
**목록/검색 페이지네이션은 offset을 기본으로 한다.** 근거:

1. **UI 요구**: 시안이 `< 1 2 3 … >` **번호 + 전체 페이지 수**를 쓴다. cursor는
   임의 페이지 점프와 전체 개수 표시가 근본적으로 어렵다 → 현재 UI와 불일치.
2. **데이터 규모/사용 패턴**: 공연 카탈로그는 수천 건 규모이고, 사용자가 수백 페이지
   깊이까지 내려가는 브라우징은 사실상 없다 → offset의 깊은 페이지 비용이 실질적 문제 아님.
3. **쓰기 특성**: 목록은 KOPIS 배치 동기화로만 바뀌어 거의 정적 → cursor의 강점(삽입 중
   페이지 밀림 안정성)이 필요 없다.
4. **단순성/재사용**: 이미 `/search`에 offset 페이지네이션이 구현돼 있고, 홈 "전체 보기"도
   이를 재사용한다. 추가 복잡도 0.

## 고려한 대안 (Alternatives)
- **cursor(keyset)**: 매우 깊은/대용량 목록, 무한 스크롤, 실시간 삽입이 잦은 피드에 유리.
  하지만 페이지 점프·total 부재로 현재 번호 페이지네이션 UI와 안 맞고, 지금 규모엔 과함.
- **무한 스크롤**: UX 취향상 제외(사용자 요청). "더 보기" 버튼도 후보였으나, 홈은 티저 +
  `/search` 유도로 정리.

## 결과/영향 (Consequences)
- 장점: UI 정합, 구현 단순, 전체 개수/페이지 점프 제공.
- 한계(수용): offset은 뒤 페이지로 갈수록 스캔+폐기 비용이 커진다(깊은 페이지 p95 저하).
  현재 규모에선 무시 가능하나, 데이터가 크게 늘면 특정 화면은 재검토 대상.

## 언제 뒤집나 (Revisit triggers)
다음 중 하나면 해당 화면만 cursor로 전환을 검토한다:
- 특정 목록이 대용량(수십만+)이고 깊은 페이지 접근이 실제로 발생.
- 화면이 "더 보기"/무한 스크롤 방식으로 바뀜(예: S07 운영 주문내역).
- 부하 테스트에서 offset 깊은 페이지 p95가 기준(performance-rules)을 초과.
  → 이때 **대용량 시드 후 offset vs cursor p95 비교를 IMP로 측정**해 근거를 남긴다.

## 참고
- 랭킹/대기열은 Redis ZSet 인덱스 기반(`ZREVRANGE`)이라 이 ADR의 SQL 페이지네이션과 별개.
