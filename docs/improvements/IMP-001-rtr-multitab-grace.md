# IMP-001 · RTR 멀티탭 세션 폭파 — grace 윈도 도입

- 슬라이스: `S01` (인증/인가)
- 날짜: 2026-06-26
- 관련 커밋: `e3613c0` (fix(auth): 멀티탭 RTR grace …)
- 유형: 보안·동시성 설계 개선 (정성 사례)

## 1. 상황 (Context)
로그인 유지를 위해 Refresh Token을 httpOnly 쿠키로 두고, 앱 로드 시
`AuthBootstrap`이 `POST /auth/refresh`로 Access를 silent 재발급하도록 구현했다.
재발급은 보안을 위해 **RTR(Refresh Token Rotation)** 방식 — 재발급 때마다 토큰을
회전시키고, 이미 회전되어 폐기된 토큰이 다시 들어오면 탈취로 간주해 토큰 패밀리를
전부 폐기(재사용 탐지)한다.

## 2. 문제 (증상)
**탭을 두 개만 열어도 로그아웃**되는 현상.
- 탭 A 로드 → `refresh()` → 회전(옛 토큰 폐기, 새 토큰 발급)
- 탭 B가 거의 동시에 로드 → 아직 **옛 토큰**을 전송
- 서버는 이를 "폐기된 토큰 재사용 = 탈취"로 오판 → **토큰 패밀리 전체 폐기**
- 결과: 두 탭 다 강제 로그아웃

> 외부 코드 리뷰에서 이 동시성 시나리오를 지적받아 확인 → 재현됨.

## 3. 원인 분석
세 가지가 겹쳐서 발생:
1. **RTR**: 매 재발급마다 토큰 회전 → 직전 토큰은 즉시 무효
2. **재사용 탐지**: 무효 토큰이 오면 보안상 패밀리 폐기
3. **무조건 회전 + 멀티탭**: 정상 사용자의 동시 요청이 "직전 토큰 제출"을 유발

즉 **보안 기능(재사용 탐지)이 정상 사용자(멀티탭)를 공격자로 오인**하는,
RTR의 알려진 함정(benign reuse)이다.

## 4. 조치 (해결)
토큰 회전에 **grace 윈도**를 도입했다. (`TokenService`)
- 회전 시 직전 토큰을 `refresh:prev:{userId}`에 **짧은 TTL(60초)**로 보관
- 재발급 요청이 **직전 토큰**이면(=grace 내 동시 요청): 폭파하지 않고
  **현재 세션을 유지한 채 새 Access만 발급**
- **grace 밖**의 토큰(2세대 이전 등)이 오면: 그때만 탈취로 간주해 패밀리 폐기

```
현재 토큰 → 정상 회전 (직전 = 현재, 현재 = 신규)
직전 토큰 → grace: 세션 유지 + 새 Access (회전 X, 폐기 X)
그 외      → REFRESH_TOKEN_REUSED + 패밀리 폐기
```

## 5. 결과
| 시나리오 | before | after |
|---|---|---|
| 멀티탭 동시 새로고침 | 전체 로그아웃 | 세션 유지 |
| 직전 토큰 재사용(grace 내) | REUSED·폐기 | 허용(새 Access) |
| 2세대 이전 토큰 재사용 | REUSED·폐기 | REUSED·폐기 (유지) |

→ 정상 동시 요청은 살리고, 진짜 탈취 탐지는 보존.

## 5-1. 후속 — 진짜 동시성 레이스를 Lua 원자화로 해결 (표면→심층)
grace로 "순차적 직전 토큰"은 막았지만, 코드 리뷰에서 **더 깊은 레이스**가 드러났다.

- **남은 문제**: `rotate`가 `GET current` → 비교 → `SET prev` + `SET 새토큰`으로
  **여러 Redis 명령에 쪼개져 비원자적**이었다. 같은 refresh로 **진짜 동시에** 두 요청이
  들어오면 둘 다 `current == 제출토큰`을 통과해 각자 새 refresh를 발급 →
  마지막 저장만 살아남고, 다른 토큰을 든 브라우저는 다음 refresh에서 REUSED를 맞을 수 있다.
- **조치**: 비교+회전 전체를 **Redis Lua 스크립트 단일 실행**으로 묶어 원자화.
  Redis가 스크립트를 직렬 실행하므로, 동시 요청 중 **첫 요청만 회전(status 1)**,
  나머지는 직전 토큰 매칭으로 **grace(status 2)** 처리되어 폭파되지 않는다.
- **검증**: 스레드 6개가 `CountDownLatch`로 동시에 같은 refresh를 보내는
  동시성 통합 테스트 추가 → 전부 200, REUSED로 인한 세션 폭파 0건.

> 정리: grace(논리적 완화) → Lua(원자성 보장)로 **표면 증상에서 근본 레이스까지** 단계적으로 보강.

## 6. 트레이드오프 / 한계
- grace 윈도를 길게 잡을수록 멀티탭 내성↑, 탈취 토큰 허용 시간↑ → 보안↓.
  **60초**로 짧게 설정해 균형.
- 더 견고히 하려면 프론트의 refresh **single-flight(탭 간 락)**를 추가해
  애초에 동시 요청 수를 줄일 수 있다(후속 개선 여지).
- grace·RTR·Lua 원자화는 알려진 표준 기법이며, 독창적 기법은 아니다.

## 7. 배운 점 (면접 답변용)
> RTR을 구현하며 멀티탭에서 직전 토큰이 재사용 공격으로 오탐되어 세션이 끊기는
> 함정을 만났습니다. 1차로 grace 윈도를 두어 정상 동시 요청은 유지하고 grace 밖의
> 폐기 토큰만 탈취로 처리했습니다. 이후 리뷰에서 비교+회전이 비원자적이라 진짜
> 동시 요청에는 레이스가 남는 걸 발견해, 해당 로직을 Redis Lua로 원자화하고
> 동시성 통합 테스트로 검증했습니다. 표면 증상에서 근본 원인까지 단계적으로
> 보강한 사례입니다.
