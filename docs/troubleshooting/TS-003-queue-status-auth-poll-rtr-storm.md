# TS-003 · 대기 상태 폴링이 토큰 회전 폭주를 유발 — 인증 경계 오배치

- 슬라이스: `S03`(대기열) · `S01`(인증 경계)
- 날짜: 2026-07-03
- 유형: 인증 경계 정합(코드)
- 관련 커밋/PR: `6803255` (fix: status 토큰-only permitAll) · PR #49
- 관련 문서: [ADR-002](../decisions/ADR-002-queue-design.md)(대기열 설계/토큰)

> 순서: 증상 → 조사 → 근본 원인 → 해결 → 재발 방지.

## 1. 증상
대기열 대기 화면은 상태를 **2초마다 폴링**(`GET /queue/status`)한다. 그런데 대기 중
프론트가 매 폴링마다 **401 → 자동 refresh(RTR 회전)** 를 반복해, 액세스·리프레시 토큰이
2초 주기로 계속 회전. 멀티탭 RTR grace와 맞물리면 불필요한 세션 처닝까지 유발할 수 있는
상태였다.

## 2. 조사
- `SecurityConfig`에서 `/queue/status`가 `.requestMatchers("/events/*/queue/**", "/queue/**").authenticated()`
  에 함께 묶여 **Bearer 인증 필수**로 처리되고 있었다.
- 폴링 호출이 만료된(또는 없는) 액세스 토큰으로 나가면 **401** → 프론트 `apiClient`의
  401 자동 재발급 로직이 **refresh 호출 → RTR 토큰 회전** → 다음 2초 뒤 또 반복.
- 정작 대기 상태는 **대기 토큰(비밀 UUID)** 만으로 식별 가능하다(발급 시 그 유저에게만
  전달된 값). Bearer가 없어도 조회에 필요한 신뢰는 토큰 자체가 제공.

## 3. 근본 원인
**인증 경계 오배치.** "비밀 토큰으로 식별되는 조회"를 굳이 Bearer 인증 뒤에 둔 탓에,
**고빈도 폴링 엔드포인트가 매 주기 인증 실패→갱신을 촉발**했다. RTR(회전형 리프레시)
환경에선 불필요한 401 하나가 세션 회전으로 증폭된다.

## 4. 해결
- `GET /queue/status`를 **permitAll**로 분리(토큰=비밀 UUID로 식별). `/queue/**` 인증 규칙보다
  **먼저 매칭**되도록 순서 배치.
- 대기열 **진입(POST)·이탈(DELETE)** 은 회원(Bearer) 유지 — 소유자 행위라 인증이 맞음.
- 설계 근거(ADR-002: 대기 토큰은 비밀값)와 일치.

```
GET  /queue/status   → permitAll  (토큰으로 식별, 고빈도 폴링)
POST /events/*/queue/token → authenticated (발급 = 회원 행위)
DELETE /queue/token  → authenticated (이탈 = 회원 행위)
```

## 5. 재발 방지
- **고빈도 폴링/SSE 엔드포인트는 인증 모델을 먼저 설계.** 비밀 토큰으로 식별 가능한 읽기는
  Bearer 뒤에 두지 말 것(폴링×만료 = 갱신 폭주).
- RTR을 쓰는 서비스에선 "불필요한 401"이 곧 세션 회전이라는 점을 인증 규칙 리뷰 체크리스트에
  포함. `/sse/**`도 같은 이유로 permitAll(토큰 경로).
