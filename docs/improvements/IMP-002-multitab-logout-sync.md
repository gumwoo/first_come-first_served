# IMP-002 · 멀티탭 로그인/로그아웃 즉시 동기화 (BroadcastChannel)

- 슬라이스: `S01` (인증/인가) · 프론트엔드
- 날짜: 2026-06-26
- 관련: [[IMP-001]](IMP-001-rtr-multitab-grace.md) (같은 "멀티탭 인증" 테마 — 서버측 토큰 회전)
- 유형: 프론트엔드 UX/보안 설계 개선 (정성 사례)

## 1. 상황 (Context)
Access Token은 탭별 메모리(Zustand)에, Refresh는 httpOnly 쿠키에 둔다.
메모리는 탭마다 독립적이라, 한 탭의 로그인/로그아웃이 다른 탭 화면에 즉시
반영되지 않는다.

## 2. 문제
한 탭에서 **로그아웃**해도 다른 탭은 화면상 여전히 로그인된 것처럼 보인다.
- 서버 차원에선 Refresh가 폐기되어 **결국엔** 풀리지만(보안은 OK),
- 다른 탭이 **API 호출/새로고침 하기 전까지는** UI가 안 바뀜
- 사용자 혼란("로그아웃했는데 왜 아직 로그인돼 보이지?") + 공용 PC 보안 체감 저하

## 3. 원인 분석
인증 UI 상태가 **탭 로컬 메모리**에 있고, 탭 간 공유는 httpOnly 쿠키(서버 인증용)
뿐이라 **즉시 전파 채널이 없음**. 서버 폐기(보안)와 화면 반영(UX)이 분리돼 있다.

## 4. 조치 (해결)
`BroadcastChannel`로 탭 간 인증 이벤트를 즉시 전파. (`features/auth/tabSync.ts`)
- 로그인/로그아웃 시 **이벤트만** 브로드캐스트(토큰은 싣지 않음 → 보안)
- 받은 탭:
  - `logout` → 즉시 `store.clear()` (화면 비로그인 전환)
  - `login` → 자신의 httpOnly 쿠키로 **스스로** silent 복원(`refresh`+`getMe`)
- 서버측 토큰 폐기(기존)는 그대로 유지 → "즉시 UX"(BroadcastChannel)와
  "확실한 무효화"(서버 폐기)를 **역할 분리**해 함께 사용.

## 5. 결과
| 행동 | before | after |
|---|---|---|
| 한 탭 로그아웃 → 다른 탭 | 행동(클릭/새로고침) 전까지 로그인된 듯 | **즉시 비로그인 전환** |
| 한 탭 로그인 → 다른 탭 | 새로고침해야 반영 | **즉시 로그인 반영**(각자 쿠키로 복원) |
| 서버 토큰 폐기 | 됨 | 됨 (유지) |

## 6. 트레이드오프 / 한계
- 토큰을 브로드캐스트하지 않으므로(이벤트만), `login` 수신 탭은 한 번 더
  `refresh`/`getMe`를 호출한다 → 약간의 추가 요청. 대신 토큰 노출 위험 없음.
- `BroadcastChannel` 미지원 환경은 자동 무시(no-op)되고, 서버 폐기로 결국 수렴.
- 표준 기법이며 독창적 패턴은 아니다.

## 7. 배운 점 (면접 답변용)
> 멀티탭에서 로그아웃이 다른 탭에 즉시 반영되지 않아, 서버 토큰 폐기(보안)와
> 별개로 BroadcastChannel로 탭 간 이벤트를 전파했습니다. 토큰을 직접 공유하지
> 않고 이벤트만 보내, 받은 탭이 자신의 httpOnly 쿠키로 스스로 복원하게 해
> 보안과 UX를 동시에 챙겼습니다. IMP-001(서버측 회전 grace)과 함께 멀티탭
> 인증 일관성을 앞단·뒷단 양쪽에서 처리한 셈입니다.

## 검증
- 수동: 같은 브라우저 탭 2~3개에서 한 탭 로그아웃 → 나머지 즉시 비로그인 확인
- (BroadcastChannel은 jsdom 미지원이라 자동화 E2E는 후속 — Playwright 멀티컨텍스트로 가능)
