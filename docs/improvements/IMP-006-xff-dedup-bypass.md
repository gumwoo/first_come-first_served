# IMP-006 · 조회수 dedup 우회 — X-Forwarded-For 신뢰 경계 도입

- 슬라이스: `S02`
- 날짜: 2026-07-01
- 유형: 정량(테스트로 측정) — 보안·정합성(경미)
- 관련 커밋: `f054967` (fix: 조회수 기록 정합성)
- 근거 테스트: `ClientIpResolverTest`

> 흐름: 문제 정의 → 증상(수치) → 가설 → 도구로 검증 → 해결 → 재측정 → 한계.

## 1. 상황 (Context)
공연 상세 진입 시 조회수를 Redis ZSet에 기록해 인기/실시간 랭킹을 만든다.
같은 IP의 반복 조회는 `view:dedup:{eventId}:{ip}` TTL 60초로 1회만 카운트한다.
코드 리뷰에서 이 dedup의 기준 IP를 어떻게 구하는지가 지적됐다.

## 2. 문제 정의 + 분류
dedup 기준 IP를 **X-Forwarded-For 헤더에서 무검증**으로 취한다.

- 계층 분류: **애플리케이션(신뢰 경계)** — 보안·경미(랭킹 한정, 금전/인증 영향 없음).
- 왜: XFF는 클라이언트가 임의로 넣을 수 있는 요청 헤더. 앞단 프록시가 세팅했다는
  보장이 없는데 그대로 신뢰.

## 3. 증상 (측정된 증거)
- 측정 방법: `ClientIpResolverTest` — 동일 peer에서 XFF만 바꾼 요청 100건의
  "해석된 IP 유니크 수"를 센다(유니크 수 = dedup을 통과하는 조회 수와 동치).
- before 수치(무검증 XFF-keyed 가정):
  | 지표 | 값 |
  |------|----|
  | 위조 XFF 100종 → 카운트되는 조회 | **100건** (dedup 완전 우회) |

## 4. 가설 (검증 전)
- 가설 1: XFF를 그대로 키로 쓰면, 공격자가 매 요청 다른 XFF를 넣어 dedup을 우회하고
  특정 공연 조회수를 무한정 부풀릴 수 있다.

## 5. 검증 (도구로 확인 → 확정 원인)
- 사용 도구: 단위 테스트로 재현. 신뢰 목록 없는 resolver에 위조 XFF 100종 주입.
- 확인 결과: 신뢰 경계가 없으면 100종이 100개의 서로 다른 IP로 해석됨 → dedup 무력.
  원인은 "직접 peer 검증 없이 XFF 신뢰"로 확정.

## 6. 조치 (해결)
`ClientIpResolver` 도입:
- 직접 접속 peer(`remoteAddr`)가 **신뢰 프록시 목록(`app.proxy.trusted`)에 있을 때만**
  XFF의 첫 홉(원 클라이언트)을 사용.
- 목록이 비면(기본) XFF를 무시하고 `remoteAddr` 사용 → 위조 무력화(안전한 기본값).
- 운영은 LB/프록시 IP를 `TRUSTED_PROXIES`로 주입.

## 7. 결과 (재측정 — 동일 테스트)
| 지표 | before | after | 변화 |
|------|--------|-------|------|
| 위조 XFF 100종(비신뢰 peer) → 카운트 | 100건 | **1건** | -99% |
| 신뢰 프록시 뒤 실제 100 클라이언트 | 100 | 100 | 유지(오탐 0) |

→ 위조는 1건으로 수렴시키되, 정상 프록시가 전달하는 실제 다수 클라이언트는 그대로 구분.

## 8. 트레이드오프 / 한계 / 다음 개선
- 신뢰 프록시 IP를 운영에서 정확히 관리해야 함(잘못 넣으면 전 사용자 IP가 프록시로 뭉침).
- 다중 프록시 체인은 첫 홉만 신뢰 — 더 엄밀히는 신뢰 hop 수만큼 우측에서 역순 파싱 필요.
- 근본적으로는 `server.forward-headers-strategy`(Tomcat RemoteIpValve internalProxies)로
  프레임워크 레벨에서 처리하는 방법도 있음(후속 고려).
- 조회수는 여전히 IP 기반이라 대규모 분산 IP 위조(봇넷)까지는 못 막음 — 랭킹 도메인
  수준에서 비용 대비 적정선.

## 9. 배운 점 (면접 답변용)
> 조회수 dedup 기준으로 X-Forwarded-For를 무검증 신뢰하면 헤더 위조로 우회돼
> 조회수가 부풀 수 있다는 리뷰를 받았습니다. 직접 접속 peer가 신뢰 프록시일 때만
> XFF를 신뢰하도록 경계를 두고, 위조 100종이 1건으로 수렴하는 것과 정상 프록시 뒤
> 100 클라이언트가 그대로 구분되는 것을 단위 테스트로 함께 검증했습니다. 보안은
> 심각도가 낮아도 신뢰 경계를 명시하는 게 운영 감각이라 판단했습니다.
