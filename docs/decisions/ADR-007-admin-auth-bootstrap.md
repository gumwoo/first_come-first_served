# ADR-007 · 관리자 인증 — /admin/** 단일 게이트 + env 부트스트랩

- 상태: Accepted
- 날짜: 2026-07-23
- 슬라이스: S07(운영/Admin)
- 관련: [[ADR-006]](주문 상태 전이), 운영 콘솔 FE, [[AGENTS]]

## 맥락
운영(S07) 화면을 붙이기 전, `/admin/**` 아래에는 개발용 엔드포인트(KOPIS 동기화·좌석 시딩)가
**인증 게이트 없이** 노출될 여지가 있었다. 여기에 대시보드·주문 조회·공연 CRUD 같은 운영 기능이
더해지면 "누가 관리자인가"와 "관리자를 어떻게 만드나"를 먼저 못 박아야 한다.

핵심 제약(프로젝트 규칙): **비밀(관리자 비밀번호 등)은 환경변수로만**. 코드·yml·채팅에 넣지 않는다.
또한 기존 인증은 JWT(access)에 `role` 클레임을 실어 `JwtAuthenticationFilter`가
`ROLE_ADMIN` 권한으로 매핑하는 구조가 이미 있다.

## 결정
1. **경로 단일 게이트**: `/admin/**` 전체를 `hasRole("ADMIN")`로 막는다. 개별 컨트롤러에
   애노테이션을 흩뿌리지 않고 `SecurityConfig` 한 곳에서 경계를 관리한다.
   ```java
   .requestMatchers("/admin/**").hasRole("ADMIN")
   ```
   기존 KOPIS 동기화·좌석 시딩도 자동으로 관리자 전용이 된다(의도된 정책 변경).
2. **관리자 계정 = env 부트스트랩(멱등)**: `AdminBootstrap`(CommandLineRunner)이 기동 시
   `ADMIN_EMAIL`/`ADMIN_PASSWORD`를 읽어 해당 이메일이 없으면 ROLE_ADMIN 1명 생성, 있으면 skip.
   미설정이면 no-op. 비밀번호는 `PasswordEncoder`로 해시 저장(원문 저장 금지).
3. **권한 매핑 재사용**: 새 권한 체계를 만들지 않고 JWT `role` 클레임 → `ROLE_ADMIN` 규약을 그대로
   사용. `hasRole("ADMIN")`이 `ROLE_ADMIN`을 기대하므로 접두사 규약이 맞물린다.
4. **FE/BE 2중 게이트**: 프론트는 `user.role === "ROLE_ADMIN"`로 화면·헤더 링크를 가리고(UX),
   **데이터 접근의 최종 판단은 서버 게이트**가 한다. FE 게이트는 편의일 뿐 신뢰 경계가 아니다.

## 고려한 대안
- **회원가입 후 승격 API**(`POST /admin/promote`): 승격 엔드포인트 자체가 권한상승 공격면이 되고,
  "누가 승격을 호출할 수 있나"라는 같은 문제를 되돌이표로 만든다 → 기각.
- **yml/코드에 관리자 계정 하드코딩**: 비밀이 저장소에 남아 유출 위험. 프로젝트 규칙 위반 → 기각.
- **별도 admin 테이블 + 별도 로그인 플로우**: 운영자 1~소수인 포트폴리오 규모에 과설계.
  기존 User/JWT/필터를 재사용하는 편이 단순하고 검증 표면도 작다 → 기각.
- **경로별 개별 `@PreAuthorize`**: 게이트가 여러 파일로 흩어져 누락 위험. 단일 `requestMatcher`가
  경계 파악·리뷰에 유리 → 기각.

## 결과 / 한계
- 통합테스트로 경계 검증: 미인증 401 / ROLE_USER 403 / ROLE_ADMIN 200(`AdminAuthIntegrationTest`).
- 관리자 미주입 환경(CI 등)에서는 부트스트랩이 no-op → 관리자 기능 비활성(정상). 비밀 노출 0.
- **한계**: 단일 role(ADMIN) 평면 권한. 세분화된 운영 권한(읽기 전용 운영자 등)은 범위 밖 —
  필요 시 role 확장 또는 권한 테이블 도입으로 뒤집는다.
- **한계**: env 부트스트랩은 "최초 1명" 시딩용. 관리자 추가·회전은 별도 운영 절차(수동 DB/후속 기능)로,
  이 ADR의 범위가 아니다.
