# Secure Coding (시큐어 코딩)

coding-standards.md §3의 심화. 태그: [H] 하네스 · [R] 리뷰 · [T] 테스트.
선착순 결제/인증 시스템이라 보안 비중이 높다. 단, 정적으로 명백한 것만 [H]로
강제하고(false positive 폭발 방지) 나머지는 리뷰/테스트로 둔다.

## 1. 시크릿 / 자격증명
- API 키·JWT secret·DB 비번 하드코딩 금지 → 환경변수/`*-local.yml`(gitignore) [H]
- 시크릿을 코드/로그/에러메시지/커밋에 노출 금지 [H 일부, R]
- KOPIS 키 등 외부 키는 백엔드에서만, 프론트 노출 금지 [H]

## 2. 인젝션
- SQL/JPQL 문자열 연결 금지 → 바인드 파라미터(`:param`) [H]
- 네이티브 쿼리에 사용자 입력 직접 삽입 금지 [H, R]
- 동적 정렬/필터 컬럼은 화이트리스트 검증 [R]

## 3. 인증 / 인가
- 비밀번호 단방향 해시(BCrypt), 평문/가역암호 금지 [R, T]
- 모든 보호 리소스에 인증 + 인가 체크 [T]
- 소유권 검증: "내 주문만 취소/조회" 등 리소스 소유자 확인 [T]
- JWT: 만료/서명검증, refresh 토큰 회전, 토큰을 URL/로그에 남기지 않기 [R]
- 로그인 실패 시 계정 존재 여부 등 과도한 정보 노출 금지 [R]

## 4. 민감정보 처리
- 비밀번호/토큰/카드번호/주민번호 로깅 금지 [H 일부, R]
- 엔티티 toString/로그에 민감 필드 노출 금지 [R]
- 응답 DTO에 불필요한 민감 필드 포함 금지 [R]

## 5. 입력 / 출력 검증
- 모든 외부 입력은 서버측 검증(@Valid) — 클라 검증 신뢰 금지 [R, T]
- 파일 업로드/외부 URL 처리 시 경로조작·SSRF 방지 [R]
- XSS: 프론트에서 dangerouslySetInnerHTML 지양, 출력 이스케이프 [H 가능, R]

## 6. 웹 보안 정책
- CORS 허용 출처 명시(와일드카드 남용 금지) [R]
- CSRF 정책 명시(토큰/세션 방식에 맞게) [R]
- 보안 헤더(HSTS 등) 고려 [R]
- 과다요청 제어(rate limit) — 선착순 어뷰징 방지 [R, T]

## 7. 의존성 / 공급망
- 허용 의존성 화이트리스트 준수(allowed-stack.yaml) [H]
- 의존성 취약점 스캔(SCA) CI 통합 [H, CI]
- 알려진 취약 버전 사용 금지 [R]

## 하네스가 정적으로 잡는 항목 (요약)
harness/backend/check.mjs 의 금지패턴 검사가 다음을 차단한다:
- 하드코딩 시크릿 의심 패턴 (`password = "..."`, `secret = "..."`, 키 리터럴)
- `printStackTrace()` / `System.out`·`System.err`
- 컨트롤러 내 try/catch (예외 삼키기)
- 문자열 연결 SQL (`"SELECT " + ...`, `createQuery("..." + ...)`)
나머지(인가/암호화/소유권/CORS 등)는 리뷰 체크리스트 + 테스트로 검증한다.
