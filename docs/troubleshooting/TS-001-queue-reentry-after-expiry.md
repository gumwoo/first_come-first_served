# TS-001 · 입장 만료 후 재예매 불가 — 죽은 토큰이 1인1토큰을 점유

- 슬라이스: `S03`(대기열)
- 날짜: 2026-07-06
- 유형: 정합성 버그(코드) — 대기열 발급 경로
- 관련 커밋/PR: `dd5c7ee` (fix: issue self-heal) · PR #59
- 관련 문서: [IMP-007](../improvements/IMP-007-token-issue-dedup.md)(1인1토큰 SET NX), [IMP-004](../improvements/IMP-004-queue-admission.md)(정원 원자 승격)

> 순서: 증상 → 재현 → 조사 → 근본 원인 → 해결 → 재발 방지.

## 1. 증상
좌석 선택 단계에서 입장(ADMITTED) 후 진행하지 않아 **입장창이 만료**된 뒤,
사용자가 공연 상세 → `예매하기`로 다시 들어와 새 순번을 받으려 하면
대기열 화면이 곧바로 **"대기 세션이 만료되었습니다"**로 떨어지고 재예매가 안 됨.
새로고침·재진입을 반복해도 동일. 유저 입장에선 "만료되면 다시 줄 서면 되는 것 아니냐"가
안 통하는 상태.

## 2. 재현
1. 대기열 입장(ADMITTED) → 좌석 선택 화면 진입.
2. 좌석을 잡지 않고 입장창(admit-ttl) 경과시켜 만료.
3. 상세 → `예매하기` → 대기열 재진입.
4. 기대: 새 순번(WAITING)으로 다시 대기. **실제: 만료 화면만 반복.**

## 3. 조사
대기열 상태는 전적으로 Redis에 있으므로 키를 추적:

- 발급 경로 `QueueService.issue()`는 `queue:user:{eventId}:{userId}`에 **SET NX**로
  1인1토큰을 예약하고, 이미 있으면 **기존 토큰을 반환**(`currentOrWaiting`).
- 입장창 만료 회수 `QueueAdmissionService.reclaim()`는 만료 토큰을 `admitexp` ZSet에서
  제거하고 `admitcount`를 감소시키며 SSE `queue.expired`만 발행.
  → **유저키(`queue:user:...`)와 토큰 메타(`queue:token:...`)는 삭제하지 않음.**
- 유저키 TTL은 `token-ttl`(기본 30분)이라, 그동안 살아있음.

즉 만료 후에도 유저키가 남아, 재진입 시 SET NX가 실패하고 **이미 죽은 토큰**이 반환된다.
그 토큰의 상태를 `statusOf`로 보면: `admit` 키는 TTL 만료로 사라졌고, `wait` ZSet에선
승격 시 이미 pop돼 없음 → **EXPIRED로 고정**. 폴링이 이를 그대로 만료로 표시.

## 4. 근본 원인
**만료 정리의 비대칭.** 슬롯 회계(admitcount)는 되돌리면서 **소유권(유저키)·토큰 메타는
남겨둬**, 1인1토큰 불변식이 "살아있는 토큰"이 아니라 "죽은 토큰"까지 붙잡았다.
`leave()`(명시적 나가기)에서만 유저키를 지우는데, **자연 만료 경로엔 그 정리가 없었다.**
결과적으로 유저키 TTL(30분)이 지나기 전까지 재예매가 원천 차단.

## 5. 해결
회수 경로를 손대는 대신, **발급 경로가 죽은 토큰을 스스로 치유(self-heal)** 하도록 수정.

- `issue()`: 유저키에 남은 기존 토큰이 **살아있으면(WAITING/ADMITTED) 재사용**(1인1토큰 유지),
  **입장 후 만료된 죽은 토큰이면 메타를 정리하고 새 토큰을 발급**해 재예매를 허용.
- `isReusable()`: `EXPIRED`로 보여도 **토큰 메타가 없으면** 아직 wait 등록 전인
  '경합 중 신규 토큰'으로 간주해 재사용(더블클릭 중복 발급 방지 — IMP-007 불변식 보존),
  **메타가 있으면** 죽은 토큰으로 판정해 폐기.

```
기존 토큰 상태     메타 존재   판정
WAITING/ADMITTED   -          재사용(1인1토큰)
EXPIRED            있음        죽은 토큰 → 폐기 후 새 발급  ← 이 버그
EXPIRED            없음        경합 중 신규 → 재사용(중복 방지)
```

프론트 변경은 불필요: 상세→예매하기로 재진입하면 컴포넌트가 remount되며 `issue()`가
새 토큰을 반환하므로 서버 수정만으로 흐름이 복구된다.

## 6. 재발 방지
- 통합 테스트 추가: `입장창이_만료된_토큰으로_재진입하면_새_순번을_받는다`
  (admit → admit-ttl 경과 → reclaim → 재진입 시 새 토큰이 WAITING으로 발급되고
  wait ZSet에 재등록됨을 검증). 회귀 시 CI에서 바로 잡힘.
- 교훈: **상태 수명이 여러 키에 나뉠 때(유저키/토큰 메타/wait/admit/admitexp), 만료·정리는
  대칭이어야 한다.** 한쪽만 되돌리면 다른 쪽이 유령 상태로 남아 불변식을 왜곡한다.

## 한계 / 남은 것
- 죽은 토큰 메타는 재진입 시점에 정리된다(재진입이 없으면 TTL로 자연 소멸). 능동적
  일괄 청소는 아니며, 현재 규모에선 불필요하다고 판단.
- 다중 서버로 확장 시에도 발급은 요청 스레드에서 일어나므로 동일 로직으로 안전
  (판정이 Redis 단일 키 관측에 기반).
