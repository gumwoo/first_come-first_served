# ADR-005 · 결제 게이트웨이 — 포트-어댑터로 격리(Mock / Toss 테스트)

- 상태: Accepted
- 날짜: 2026-07-07
- 슬라이스: S05(주문·결제)
- 관련: [[order-payment]], [[ADR-006]]

## 맥락
결제는 외부 PG(토스페이먼츠 등)와 통신해야 한다. 그런데 서비스 로직 안에서 PG SDK를 직접
호출하면: (1) 테스트마다 실제 네트워크·비밀키가 필요해 느리고 불안정(flaky), (2) 외부 장애가
우리 테스트/CI를 깨뜨리고, (3) PG 교체 시 도메인 코드가 흔들린다.
포폴 관점에선 "실 PG 연동을 안 하고 시뮬레이션했다"로 보이는 것도 피하고 싶다.

## 결정
1. **`PaymentGateway` 인터페이스(포트)** 로 결제 외부 의존을 격리한다.
   `approve(승인)` / `issueVbank(가상계좌)` / `verifyWebhook(서명검증)`.
2. 구현체 2개:
   - **MockPaymentGateway** — 테스트/E2E/CI. 금액·플래그로 성공/실패를 **결정론적**으로. 네트워크 X.
   - **TossPaymentGateway** — 로컬/데모. 토스 **테스트모드**(테스트키) 승인 API + 웹훅 서명검증.
3. 설정 `payment.gateway=mock|toss`로 주입. **비밀키(TOSS_SECRET_KEY 등)는 환경변수로만.**
4. 자동 테스트(통합/E2E)는 항상 Mock → 결정론·독립성. 실 연동은 로컬/데모에서 Toss로 시연.

## 고려한 대안
- **순수 목만**: 가장 단순하나 외부 연동 스킬 미노출 + "제일 어려운 부분 회피"로 보일 수 있음.
- **PG SDK 직접 호출(어댑터 없음)**: 테스트 flaky·외부 장애 전파·교체 비용 → 기각.
- **아임포트/포트원(어그리게이터)**: 여러 PG 추상화 이점은 있으나, 포트 인터페이스는 우리가 이미
  두므로 어댑터만 갈아끼우면 됨(향후 옵션).

## 결과 / 한계
- 테스트는 Mock으로 안정적, 실 연동은 어댑터 교체로 확보(도메인 코드 불변).
- 토스 **웹훅 로컬 수신엔 공개 URL(ngrok 등)** 이 필요 — 실행 이슈이며 설계엔 무관.
- 웹훅은 서명검증 통과분만 신뢰(위조 방지). 멱등 처리는 [[ADR-006]]·`payments.idempotency_key`.
