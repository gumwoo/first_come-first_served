# ADR-003 · 재고(좌석) 원자성은 DB 조건부 UPDATE로

- 상태: Accepted
- 날짜: 2026-07-02
- 슬라이스: S04(좌석·재고)
- 관련: [[seat]], IMP-003(oversell)

## 맥락
선착순에서 "재고 1석에 2명 동시 선점"의 **초과판매(oversell)** 를 0으로 만들어야 한다.
좌석 상태는 우리 DB(seats.status)에 있고, 좌석 **식별**(어느 좌석인지)이 필요하다.

## 결정
좌석 선점을 **DB 조건부 UPDATE**로 원자화한다.
```sql
UPDATE seats SET status='HELD', version=version+1, updated_at=now()
WHERE id IN (:seatIds) AND status='AVAILABLE'
```
→ **영향 행 수 == 요청 좌석 수** 이면 성공, 아니면 롤백 + `SOLD_OUT`.
행 락으로 동시 트랜잭션 중 하나만 통과 → oversell 원천 차단. (상태가 DB에 있으니 원자성도 DB에서 — S03의 계층 원칙 일관.)

## 고려한 대안 (모두 정확성은 달성 가능, 갈리는 건 적합성)
- **naive check-then-act** (`SELECT AVAILABLE → UPDATE`): 동시 실행 시 이중 선점 → **배제**(문제 그 자체).
- **비관락(SELECT ... FOR UPDATE)**: 정확하나 좌석 행을 미리 잠가 **락 경합·데드락** 위험, 코드 복잡.
- **낙관락(@Version)**: 충돌 시 **재시도 루프** 필요, 다좌석 동시 선점에서 재시도 폭증.
- **Redis 등급 카운터 원자 차감**: 초고 TPS엔 유리하나 **좌석 식별**을 DB와 이중 관리(정합성 복잡).
  → 초대형 트래픽 화면(특정 인기 공연)만 후속 검토.
- **조건부 UPDATE(채택)**: 단일 문장이 곧 원자 연산 + 좌석 식별 그대로 + 재시도 불필요 → 단순·정확.

## 결과 / 한계
- 데모 규모(등급별 수십 석)에 최적. 좌석맵이 수만 석급이면 UPDATE 대상 인덱싱/파티셔닝 고려.
- `@Version`은 보조로 유지(다른 갱신 경로 대비).

## 언제 뒤집나
- 특정 공연이 초고 TPS로 DB가 병목 → 그 이벤트만 Redis 재고 카운터로 전환(핫패스 분리),
  이때 대용량 시드 후 조건부UPDATE vs Redis p95 비교를 IMP로 측정.

## 참고
- 대기열(S03)이 앞단에서 트래픽을 정원 단위로 직렬화하므로, 재고에 도달하는 동시성은 이미 제한됨.
