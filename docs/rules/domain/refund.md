# Domain · S06 취소 / 환불

- 취소는 `PAID` 상태에서만 가능. 그 외 → `REFUND_NOT_ALLOWED`. [T]
- 환불액 = 결제액 − 기간별 수수료. (수수료 정책은 이벤트/기간별 정의) [R]
- 환불 완료 시 좌석 재고 복구 + 상태 `REFUNDED`. [T]

## 왜 `PAID → CANCELLED → REFUNDED` 2단계인가
취소를 `PAID → REFUNDED` 한 방에 처리하지 않고 **CANCELLED를 중간 상태로** 둔다. 이유는 두 사건이
본질적으로 다르기 때문이다:
- **CANCELLED = 예매 무효화** — 좌석이 반환되고 이 예매는 더 이상 유효하지 않다(사용자 관점의 "취소됨").
- **REFUNDED = 실제 환불 완료** — 돈이 실제로 돌아갔다(PG 정산 관점).

카드 취소는 즉시라 지금은 한 트랜잭션에서 두 전이를 이어 처리하지만, **무통장·일부 PG는 환불 정산이
비동기(지연)** 다. 이때 "취소는 됐는데 환불은 진행 중"을 표현하려면 두 상태가 분리돼 있어야 한다.
즉 2단계는 **비동기 환불 정산 대비 + 감사추적**이 목적이다(현재 구현은 동기 처리 — 확장 대비 선제 분리).
전이는 각각 조건부 UPDATE로 원자화한다([[ADR-006]]): `PAID→CANCELLED`가 1행이면 이 요청이 취소의
주인, 이어서 `CANCELLED→REFUNDED`. 이중 환불은 `refunds.idempotency_key` UNIQUE로 차단([[IMP-009]]).
