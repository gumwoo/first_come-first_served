# IMP-005 · KOPIS 동기화 커버리지 — 단일 페이지/30일 → 청크·페이지 전량 수집

- 슬라이스: `S02`
- 날짜: 2026-07-01
- 유형: 정량(수치) — after는 실제 재sync 후 확정
- 관련 커밋: `a8eb4a0` (feat: KOPIS 수집 확장)
- 벤치 파일: `benchmarks/kopis-sync-before.json` → `benchmarks/kopis-sync-after.json` (재sync 후 기록)

> 흐름: 문제 정의 → 증상(수치) → 가설 → 도구로 검증 → 해결 → 재측정 → 한계.

## 1. 상황 (Context)
S02 공연 카탈로그는 KOPIS OpenAPI에서 공연 목록을 동기화한다. 메인 슬라이더·랭킹·
검색 페이지네이션이 "데이터가 풍부해야" 실질적으로 살아나는데, 화면을 붙여 보니
공연 수가 항상 100건에서 멈춰 있었다.

## 2. 문제 정의 + 분류
동기화가 **가용 데이터의 일부만 수집**한다(뒤 페이지·30일 이후 공연 누락).

- 계층 분류: **애플리케이션 코드**(동기화 로직) — 외부 API 자체 문제 아님.
- 왜: KOPIS는 더 많은 데이터를 제공하는데, 우리 `sync()`가 한 번만/좁게 호출.

## 3. 증상 (측정된 증거)
- 측정 방법: `POST /admin/sync/kopis` 실행 후 `events` count / 응답 `synced`.
- before 수치:
  | 지표 | 값 |
  |------|----|
  | 동기화 건수(synced) | **100건에서 상한** |
  | 조회 기간 | 오늘~+30일만 |
  | 페이지 | cpage=1 (1페이지) |

## 4. 가설 (검증 전)
- 가설 1: `rows=100` + `cpage=1` 고정이라 101번째부터 안 들어온다(페이지 미순회).
- 가설 2: 기간이 `+30일`로 짧아 그 이후 공연이 누락된다.

## 5. 검증 (도구로 확인 → 확정 원인)
- 사용 도구: 코드/요청 파라미터 확인 + KOPIS 명세(단일 조회 31일 제한).
- 확인 결과: `KopisSyncService.sync()`가 `fetchList(오늘, +30일, cpage=1, rows=100)`
  단일 호출 → **페이지 미순회 + 기간 협소** 둘 다 원인으로 확정.

## 6. 조치 (해결)
- `fetchListAll`: 반환 건수 < rows일 때까지 cpage 순회(`max-pages` 상한).
- `sync()`: 오늘~+`kopis.sync.days`(기본 90)를 **31일 청크로 분할**, 청크별 전량 수집 후
  upsert(kopis_id 멱등 → 청크 경계 중복 안전).
- 수집량을 `kopis.sync.*`로 외부화(코드 수정 없이 조절).

## 7. 결과 (재측정 — 재sync 후 확정)
- 측정 방법: 동일하게 `POST /admin/sync/kopis` → `events` count 비교.
- after 수치:
  | 지표 | before | after | 변화 |
  |------|--------|-------|------|
  | 동기화 건수 | 100(상한) | _재sync 후 기입_ | _+N_ |
  | 조회 기간 | +30일 | +90일(3청크) | 3배 |

> after는 실제 KOPIS 키로 재sync한 뒤 수치를 채우고 benchmarks JSON을 커밋한다.

## 8. 트레이드오프 / 한계 / 다음 개선
- 외부 호출 횟수↑(청크 × 페이지) → 동기화 소요시간·KOPIS 부하 증가. best-effort로
  일부 실패 시 수집분만 반영. `max-pages`로 폭주 방지.
- 다음: 동기화 소요시간도 측정해 배치 성능 IMP로 확장 가능(청크 병렬화 여지).
- 지역/장르 등 KOPIS 분류를 그대로 신뢰 — 분류 편차는 별도 보정 이슈.

## 9. 배운 점 (면접 답변용)
> 외부 API 연동에서 "동작한다"와 "데이터를 온전히 수집한다"는 다른 문제였습니다.
> 공연 수가 100에서 멈춘 증상을 보고 페이지 미순회와 기간 협소를 원인으로 좁힌 뒤,
> API의 31일 제한을 청크 분할로 우회하고 페이지를 끝까지 순회하도록 바꿨습니다.
> 대신 호출 횟수가 늘어 best-effort와 페이지 상한으로 안정성을 지켰습니다.
